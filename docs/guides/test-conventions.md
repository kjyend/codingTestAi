# Test Conventions

이 문서는 이 저장소에서 코딩테스트 문제의 테스트 코드를 얼마나, 어떤 기준으로 작성할지 정리한 문서다.

목표는 테스트를 과하게 복잡하게 만들지 않으면서도, 실제 코딩테스트에서 자주 나오는 큰 실수를 걸러내는 것이다.

## Basic Rule

모든 문제는 기본적으로 최소 3개의 테스트를 가진다.

기본 구성:

- 예제 테스트 1개
- 범위 또는 시간복잡도 관련 테스트 1개
- 함정 또는 반례 테스트 1개

함정이 많거나 조건 분기가 많으면 테스트를 더 추가한다.

## Test Class Rule

테스트 클래스 이름은 항상 `SolutionTest`를 사용한다.

파일 이름도 아래를 따른다.

```text
SolutionTest.java
```

이 규칙의 목적은 문제마다 테스트 파일 구조가 흔들리지 않게 하는 것이다.

## Minimum Test Set

### 1. Sample Test

문제에 나온 예제 입력과 예제 출력을 그대로 검증한다.

역할:

- 기본 정답 여부 확인
- 입력 파싱과 출력 형식의 큰 문제 확인

### 2. Limit Test

입력 크기나 반복 구조 때문에 비효율 풀이가 드러날 수 있는 경우를 점검한다.

예:

- `N`이 커질 때 `O(N^2)` 풀이가 위험한 문제
- 정렬, 탐색, 그래프 순회 횟수가 성능에 영향을 주는 문제

역할:

- 시간초과 가능성이 있는 접근을 한 번 의심하게 만든다
- 문제의 핵심 제약을 놓치지 않았는지 확인한다

주의:

- 이 테스트 하나로 성능을 완전히 증명하려고 하지 않는다
- 시간복잡도 판단은 반드시 `notes.md`에도 함께 남긴다

### 3. Trap Test

자주 틀리는 반례나 문제의 함정을 확인한다.

예:

- 중복값
- 최소 크기 입력
- 도달 불가 케이스
- 모두 같은 값
- 경계 인덱스
- 정렬 기준이 헷갈리는 경우

역할:

- 구현 실수와 놓치기 쉬운 조건을 잡는다
- "정답처럼 보이지만 틀린 풀이"를 걸러낸다

## When To Add More Tests

아래 경우에는 테스트를 더 만든다.

- 조건 분기가 많은 문제
- 함정이 여러 개 보이는 문제
- 반례가 명확하게 여러 개 떠오르는 문제
- 같은 알고리즘이어도 구현 실수 포인트가 많은 문제

즉, 기본은 3개지만 문제 성격상 필요하면 늘릴 수 있다.

## Naming Rule

테스트 메서드 이름은 역할이 드러나게 작성한다.

권장 예:

- `sampleCase`
- `limitCase`
- `trapCase`

조금 더 설명형으로 써도 된다.

예:

- `sampleCase_returnsExpectedAnswer`
- `limitCase_handlesLargeInput`
- `trapCase_returnsExpectedAnswer`

핵심은 문제마다 너무 다른 스타일로 흔들리지 않는 것이다.

## Notes Connection

테스트만으로 모든 실패 가능성을 다 잡을 수는 없다.

그래서 `notes.md`에는 아래를 함께 남긴다.

- 시간초과 가능성이 있었던 접근
- 이 문제의 함정
- 반례와 실수 포인트

즉:

- `SolutionTest.java`는 실제 검증용
- `notes.md`는 사고 과정과 위험 요소 복기용

## Final Rule

이 저장소의 테스트는 "많이 만드는 것"보다 "핵심 실수를 잡는 것"을 우선한다.

기본 원칙은 아래 한 줄로 요약할 수 있다.

`모든 문제는 최소 3개의 테스트를 가진다. 예제, 범위, 함정 테스트를 기본으로 하고, 함정이 많은 문제는 테스트를 추가한다.`
