# Commit Message Conventions

이 문서는 이 저장소에서 사용할 커밋 메시지 규칙을 정리한다.

## Basic Format

- 기본 형식은 `type : 관련 내용 한글`로 작성한다.
- `type`은 영어 소문자로 고정한다.
- 콜론 앞뒤 공백은 `type : 내용` 형태로 통일한다.
- 내용은 짧은 한글 문장 또는 구로 작성한다.
- 가능하면 하나의 커밋에는 하나의 목적만 담는다.

예시:

- `feat : 계단 오르기 DP 풀이 추가`
- `test : 계단 오르기 반례 테스트 추가`
- `refactor : 점화식 계산 로직 메서드 분리`

## Recommended Types

- `feat`: 새로운 문제 풀이, 새 템플릿, 새 topic 문서나 reference 문서를 추가할 때 사용한다.
- `test`: 테스트 케이스를 추가하거나 수정하고, 반례 검증을 강화할 때 사용한다.
- `refactor`: 동작은 유지한 채 메서드 분리, 이름 개선, 중복 제거처럼 구조를 개선할 때 사용한다.
- `fix`: 오답, 경계값 처리, 입력 처리, 인덱스 오류 같은 실제 문제를 수정할 때 사용한다.
- `docs`: README, guide, template, notes, problem 문서를 수정하거나 설명을 보강할 때 사용한다.
- `chore`: 빌드 설정, 폴더 정리, 도구 설정, workflow 같은 부가 작업에 사용한다.
- `perf`: 같은 정답을 더 효율적으로 계산하도록 시간복잡도나 공간복잡도를 개선할 때 사용한다.
- `style`: 동작 변화 없이 공백, 줄바꿈, import 정렬, 주석 스타일만 정리할 때 사용한다.

## Type Selection Rules

- 새 문제 풀이를 처음 추가하면 `feat`를 우선 사용한다.
- 테스트만 추가하거나 수정하면 `test`를 사용한다.
- 문제 풀이의 결과는 같고 구조만 좋아졌다면 `refactor`를 사용한다.
- 정답이 틀렸거나 반례에서 실패하던 문제를 고쳤다면 `fix`를 사용한다.
- 문서만 바꿨다면 `docs`를 사용한다.
- 성능 개선이 핵심 목적이면 `perf`를 사용한다.
- 여러 성격이 섞였으면 가장 핵심 목적 하나를 고른다.

## Repository Examples

- `feat : 계단 오르기 DP 풀이 추가`
- `feat : DP topic 공통 체크리스트 문서 추가`
- `test : 계단 오르기 큰 입력 테스트 추가`
- `refactor : DP 상태 전이 계산 메서드 분리`
- `fix : 계단 수가 1일 때 base case 오류 수정`
- `docs : 문제 풀이 워크플로 문서 보강`
- `chore : Gradle 기본 설정 추가`
- `perf : 누적 합 계산으로 시간복잡도 개선`

## Notes

- 커밋 메시지에 불필요한 영어 설명을 길게 붙이지 않는다.
- 가능하면 커밋 하나만 봐도 변경 의도가 떠오르도록 작성한다.
- 여러 파일을 수정했더라도 목적이 하나면 같은 타입을 사용한다.
