# Problem Reading Signals

문제를 읽을 때 바로 코드를 쓰기보다, 문제 문장에서 어떤 신호가 보이는지 먼저 분류한다.

이 문서는 "어떤 표현이 나오면 어떤 알고리즘이나 자료구조를 떠올려야 하는가"를 정리하기 위한 가이드다.

중요한 점은 키워드만 보고 기계적으로 결정하지 않는 것이다. 같은 "최소"라도 가중치 유무, 상태 공간 구조, 제약 크기에 따라 접근이 달라진다.

## Scope

이 저장소는 일반적인 코딩테스트를 대부분 커버할 수 있는 알고리즘 범위를 목표로 한다.

기본적으로 아래 범주를 포함한다.

- 구현
- 정렬
- 스택, 큐, 덱
- 해시
- BFS, DFS
- 백트래킹
- 그리디
- 이분 탐색
- 투 포인터, 슬라이딩 윈도우
- 누적합
- DP
- 다익스트라
- 플로이드 워셜
- 크루스칼
- 프림
- 유니온 파인드
- 위상 정렬
- 최소 신장 트리
- 최단 경로

너무 대회형이거나 특수한 고급 알고리즘은 기본 범위에서 제외하고, 필요할 때만 별도로 확장한다.

## First Reading Checklist

문제를 읽을 때 먼저 아래를 체크한다.

1. 입력 크기가 얼마나 큰가
2. 정답이 값 하나인지, 경로인지, 경우의 수인지
3. 그래프, 격자, 배열, 문자열 중 무엇으로 볼 수 있는가
4. 최소, 최대, 최단, 횟수, 비용 같은 목표가 있는가
5. 한 번만 계산하면 되는가, 아니면 여러 쿼리를 처리해야 하는가
6. 모든 경우를 시도해도 되는 크기인가

## Complexity Sense

정확한 기준은 문제마다 다르지만, 처음 판단할 때 아래 감각을 기본으로 둔다.

- `N`이 매우 크면 `O(N^2)` 이상은 먼저 의심한다.
- `N`이 `10^5` 이상이면 보통 `O(N log N)` 또는 `O(N)` 쪽을 먼저 검토한다.
- `N`이 작으면 완전탐색, 백트래킹, 비트마스킹 같은 방법도 가능하다.
- 정점 수와 간선 수가 함께 주어지면 `O(V + E)`인지 `O(E log V)`인지 먼저 따져본다.
- 여러 번의 구간 질의가 있으면 매번 다시 계산하지 않는 방식이 필요한지 본다.
- 상태 수가 작고 단계별 이동이면 BFS/DP를 의심한다.

핵심은 "문장 키워드"보다 "입력 범위와 시간복잡도"를 먼저 맞추는 것이다.

## Signals And Likely Directions

### BFS

아래와 같은 표현이 보이면 BFS를 먼저 의심한다.

- 최소 이동 횟수
- 가장 적은 연산 횟수
- 시작점에서 도착점까지 최단 단계 수
- 격자에서 상하좌우 이동
- 간선 가중치가 없거나 모두 동일

예:

- "가장 적은 버튼 클릭 수"
- "최소 칸 수로 도착"
- "한 번 이동할 때 비용이 모두 같다"

주의:

- "최단"이라는 말이 있어도 간선 가중치가 다르면 BFS가 아니라 다익스트라를 봐야 한다.
- 현재 문제를 BFS로 풀었더라도, 가중치가 생기거나 비용이 달라지면 접근이 바뀔 수 있다.

### DFS / Backtracking

아래와 같은 표현이 보이면 DFS 또는 백트래킹을 본다.

- 가능한 모든 경우 탐색
- 조합, 순열, 부분집합
- 작은 입력 크기
- 경로를 직접 구성해야 함

예:

- "모든 가능한 순서를 구하라"
- "N이 작으므로 완전탐색 가능"

팁:

- 현재는 입력이 작아서 DFS/백트래킹이 가능하지만, 범위가 커지면 DP나 그리디로 바뀔 수 있다.

### PriorityQueue / Heap

아래 표현은 우선순위 큐를 떠올리게 한다.

- 가장 작은 값 또는 가장 큰 값을 계속 꺼내야 함
- 상위 K개 유지
- 작업 우선순위 처리
- 현재 최솟값 또는 최댓값을 반복적으로 갱신
- 가중치가 있는 최단 경로

예:

- "가장 빨리 끝나는 작업부터 선택"
- "매 단계마다 최소 비용 간선 선택"
- "중간값, 상위 K개, 최대 힙/최소 힙 관리"

팁:

- "매번 가장 작은 값" 또는 "매번 가장 큰 값"을 빠르게 꺼내야 하면 배열보다 `PriorityQueue`를 먼저 본다.

### Dijkstra

아래 표현이면 다익스트라를 검토한다.

- 최소 비용
- 최단 시간
- 가중치가 있는 그래프 최단 경로
- 한 지점에서 다른 지점들까지 최소 거리

보통 `PriorityQueue`와 같이 생각한다.

주의:

- 음수 가중치가 있으면 다익스트라가 맞지 않을 수 있다.
- 모든 정점 쌍의 최단 거리가 필요하면 플로이드 워셜도 후보가 된다.

### Dynamic Programming

아래 표현이면 DP를 떠올린다.

- 경우의 수를 세기
- 최적값 누적
- 이전 상태를 재사용
- 부분 문제의 답으로 전체 답 구성
- 같은 계산이 반복될 것 같음

예:

- "i번째까지 고려했을 때의 최댓값"
- "앞의 결과를 이용해 다음 상태 계산"

팁:

- 완전탐색이 떠오르더라도 같은 상태가 반복되면 DP로 바꿀 수 있는지 확인한다.

### Greedy

아래 상황은 그리디를 의심한다.

- 매 순간의 최선 선택이 전체 최선으로 이어질 근거가 있음
- 정렬 후 순서대로 처리 가능
- 구간 선택, 회의실 배정, 최소 동전 문제의 특수 케이스

주의:

- 그리디는 직관만으로 결정하지 말고, 왜 항상 맞는지 설명 가능한지 확인해야 한다.
- 지금 문제에서 그리디가 맞아도, 제약이 살짝 바뀌면 DP가 필요한 경우가 많다.

### Binary Search

아래 표현은 이분 탐색으로 연결될 수 있다.

- 정답의 범위를 기준으로 탐색 가능
- 어떤 값이 가능하면 그보다 큰 값 또는 작은 값도 가능
- 최댓값의 최소화, 최소값의 최대화

예:

- "가능한 최소 시간"
- "조건을 만족하는 최대 길이"

팁:

- 배열에서 찾는 이분 탐색뿐 아니라, 정답 자체를 탐색하는 파라메트릭 서치도 함께 떠올린다.

### Two Pointers / Sliding Window

아래 표현이면 투 포인터나 슬라이딩 윈도우를 본다.

- 연속 부분 구간
- 부분합
- 문자열에서 연속 구간 조건
- 구간을 늘리고 줄이며 조건 유지

주의:

- 음수가 섞인 부분합 문제는 단순 투 포인터가 바로 안 될 수 있다.
- 이번 문제를 큐나 배열 순회로 풀었더라도, "연속 구간" 조건이 핵심이면 투 포인터가 더 자연스러울 수 있다.

### Prefix Sum

아래 표현이면 누적합을 생각한다.

- 구간 합
- 여러 번의 구간 질의
- 2차원 구간 합

팁:

- 매 요청마다 구간을 다시 더하면 느리므로, 질의가 많을수록 누적합을 먼저 본다.

### Union-Find

아래 표현이면 유니온 파인드를 떠올린다.

- 같은 집합인지 확인
- 연결 여부를 빠르게 판단
- 간선 추가에 따른 그룹 병합

팁:

- 그래프를 직접 순회하지 않고도 연결 상태만 빠르게 관리해야 하면 유니온 파인드가 강력하다.

### Floyd-Warshall

아래 표현이면 플로이드 워셜을 검토한다.

- 모든 정점에서 모든 정점까지의 최단 거리
- 정점 수가 비교적 작음
- 경유지를 거치는 최단 거리 정보가 필요함

주의:

- 정점 수가 크면 `O(V^3)`이 부담될 수 있으니 범위를 먼저 본다.

### Kruskal / Minimum Spanning Tree

아래 표현이면 크루스칼이나 MST를 떠올린다.

- 모든 정점을 최소 비용으로 연결
- 도로, 네트워크, 전선 연결 비용 최소화
- 사이클 없이 전체를 연결

팁:

- 간선 기준 정렬 후 선택이 자연스럽고, 연결 여부 판단이 중요하면 크루스칼 + 유니온 파인드를 본다.

### Topological Sort

아래 표현이면 위상 정렬을 본다.

- 선행 조건
- 먼저 해야 하는 작업
- 순서 제약이 있는 작업들

## Per-Problem Tips

문제마다 아래 내용을 꼭 확인한다.

- 이 문제의 핵심 포인트는 무엇인가
- 입력 크기를 안 보면 시간초과가 나는가
- 지금 선택한 알고리즘 말고도 가능한 후보가 있는가
- 조건이 달라지면 어떤 알고리즘으로 바뀌는가

예:

- "이번 문제는 N이 작아서 완전탐색이 가능하다"
- "이번 문제는 최단 거리지만 가중치가 없으므로 BFS가 맞다"
- "이번 문제는 큐 기반 탐색으로 풀었지만, 조건이 연속 구간이었다면 투 포인터가 더 적합했을 수 있다"

## What To Write In notes.md

`notes.md`는 정답 코드 파일이 아니라, 왜 이 풀이를 선택했는지와 무엇을 배웠는지를 남기는 복기 파일이다.

문제를 풀 때 `notes.md`에는 아래 내용을 남긴다.

- 문제에서 보이는 핵심 신호
- 사용 가능한 알고리즘 후보
- 최종 선택한 알고리즘
- 그 알고리즘을 선택한 이유
- 입력 크기와 시간복잡도 판단
- 이 문제의 핵심 포인트
- 조건이 달라졌다면 바뀌는 풀이
- 반례
- 실수 포인트
- 다음에 비슷한 문제가 나오면 체크할 것

## Final Reminder

키워드는 힌트일 뿐 정답이 아니다.

항상 아래 순서로 생각한다.

1. 문제 구조 파악
2. 입력 크기 확인
3. 가능한 알고리즘 후보 나열
4. 후보별 시간복잡도 비교
5. 가장 적절한 풀이 선택
