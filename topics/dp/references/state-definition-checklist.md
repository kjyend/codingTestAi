# DP State Definition Checklist

DP 문제를 풀 때 가장 먼저 흔들리는 부분은 보통 "상태를 어떻게 잡을 것인가"다.

이 문서는 DP 상태를 정의할 때 빠르게 점검하는 체크리스트다.

## First Question

먼저 아래 한 줄을 말로 설명할 수 있어야 한다.

`dp[...]는 무엇을 의미하는가?`

예:

- `dp[i]` = i번째까지 고려했을 때의 최댓값
- `dp[i]` = i에 도달하는 방법의 수
- `dp[i][j]` = i번째까지 봤고, j 상태일 때의 최적값

이 문장이 애매하면 코드도 흔들릴 가능성이 높다.

## Checklist

1. 상태가 문제의 핵심 정보를 모두 담고 있는가
2. 상태가 너무 많은 정보를 담아서 비효율적이지 않은가
3. 현재 상태를 이전 상태로부터 계산할 수 있는가
4. base case를 자연스럽게 정의할 수 있는가
5. 답이 어느 상태에 저장되는지 분명한가

## Common Shapes

### 1. One-Dimensional DP

형태:

- `dp[i]`

주로 쓰는 경우:

- i번째까지 고려
- 길이 i일 때의 답
- 위치 i에 도달했을 때의 답

### 2. Two-Dimensional DP

형태:

- `dp[i][j]`

주로 쓰는 경우:

- i번째까지 보면서, 추가 조건 j를 함께 관리할 때
- 두 문자열, 두 수열, 두 좌표를 함께 볼 때
- 개수, 합, 상태 여부를 추가 차원으로 둘 때

### 3. Choice-Based State

형태:

- 선택함 / 선택하지 않음
- 이전 상태의 사용 여부

주로 쓰는 경우:

- 연속 선택 제한
- 특정 조건에 따라 다음 선택 가능 여부가 달라질 때

## Warning Signs

아래 상황이면 상태 정의를 다시 봐야 한다.

- 상태 설명을 한 문장으로 못 하겠다
- 전이할 때 예외 분기가 너무 많다
- 같은 의미의 상태가 여러 방식으로 중복된다
- 메모리 크기가 지나치게 커진다
- 답이 어디 저장되는지 끝까지 헷갈린다

## Practical Rule

처음에는 가장 단순한 상태부터 시도한다.

- `dp[i]`로 가능한가
- 안 되면 어떤 정보가 더 필요한가
- 그때 차원을 하나 추가할지, 상태 의미를 바꿀지 본다

DP를 어렵게 만드는 건 점화식보다 상태 정의인 경우가 많다.
