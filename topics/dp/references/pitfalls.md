# DP Pitfalls

이 문서는 DP 문제를 풀 때 자주 하는 실수를 정리한 문서다.

## 1. 상태 정의 없이 코드부터 쓰기

가장 흔한 실수다.

`dp[i]`가 무엇인지 분명하지 않으면 점화식도 흔들리고, 디버깅도 어려워진다.

## 2. Base Case를 대충 두기

점화식이 맞아도 초기값이 틀리면 전체 결과가 틀린다.

특히 아래를 조심한다.

- 첫 번째 원소
- 빈 상태
- 시작 위치
- 길이 0 또는 1

## 3. 최댓값 문제와 경우의 수 문제를 혼동하기

둘 다 DP처럼 보여도 저장하는 값의 의미가 완전히 다르다.

- 최댓값 문제: 비교가 중요
- 경우의 수 문제: 누적 합산이 중요

## 4. 시간복잡도보다 점화식 예쁨만 보기

점화식이 보여도 전이 비용이 크면 전체 시간복잡도가 위험할 수 있다.

항상 아래를 본다.

- 상태 수
- 상태 하나당 전이 비용

## 5. 메모리 크기를 계산하지 않기

2차원, 3차원 DP는 금방 메모리가 커진다.

필요하면 아래를 고민한다.

- 상태 축소
- 1차원 압축
- 필요한 이전 행만 유지

## 6. 인덱스 기준 흔들리기

0-based와 1-based를 섞어 쓰면 실수하기 쉽다.

배열 크기, 반복 범위, base case를 한 번에 같이 정리하는 것이 좋다.

## 7. DP가 아닌데 억지로 DP로 보려고 하기

모든 누적 문제를 DP로 풀 필요는 없다.

아래 알고리즘이 더 자연스러울 수 있다.

- 그리디
- 투 포인터
- 누적합
- BFS
- 이분 탐색

## Final Check

코드를 쓰기 전에 아래를 확인한다.

1. 상태 정의가 한 문장으로 되는가
2. base case가 자연스러운가
3. 전이가 맞는가
4. 시간복잡도가 범위 안에 들어오는가
5. 더 단순한 알고리즘은 없는가
