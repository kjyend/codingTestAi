# DP

## Topic Overview

- Topic Name: DP
- 이 topic을 한 문장으로 설명하면: 같은 계산이 반복되거나, 이전 상태의 답을 이용해 현재 답을 만들 수 있을 때 사용하는 누적 최적화 기법

## Why This Topic Matters

DP는 코딩테스트에서 매우 자주 나오고, 쉬운 문제부터 꽤 고민이 필요한 문제까지 폭넓게 연결되는 대표 유형이다.

이 저장소에서 DP는 아래 목표로 학습한다.

- DP를 떠올려야 하는 문제 신호를 익힌다
- 상태를 어떻게 정의해야 하는지 연습한다
- 점화식을 어떻게 세우는지 익힌다
- 범위에 따라 완전탐색에서 DP로 전환해야 하는 상황을 판단한다

이 topic은 특히 `중하 ~ 중상` 난이도에서 유형을 가장 잘 알려주는 문제에 집중하는 것을 목표로 한다.

즉, 너무 쉬운 피보나치 수준에 머무르지 않고, 그렇다고 대회형 고난도 DP까지 바로 가지도 않는다.

## Core Concept

- 기본 개념: 부분 문제의 답을 저장하고 재사용해서 전체 문제를 해결한다
- 어떤 상황에서 이 topic을 떠올려야 하는가:
  - 같은 계산이 반복될 것 같을 때
  - 현재 선택이 이전 결과에 의존할 때
  - 경우의 수, 최댓값, 최솟값을 누적해서 계산할 수 있을 때
  - 완전탐색은 가능해 보이지만 중복 상태가 너무 많을 때
- 문제에서 자주 보이는 신호:
  - 경우의 수를 구하라
  - 최댓값 또는 최솟값을 구하라
  - i번째까지 고려했을 때
  - 앞의 결과를 이용해서 현재 결과를 만든다
  - 선택한다 / 선택하지 않는다

## What To Check First

문제를 보면 먼저 아래를 확인한다.

1. 완전탐색을 하면 같은 상태를 여러 번 다시 계산하게 되는가
2. 상태를 인덱스, 개수, 위치, 합, 선택 여부 등으로 표현할 수 있는가
3. 현재 답이 이전 몇 개 상태의 답으로 계산되는가
4. 답이 최댓값, 최솟값, 경우의 수 중 무엇인가
5. `N` 범위를 봤을 때 완전탐색은 너무 크고, `O(N)` 또는 `O(N^2)` 정도는 가능한가

## Complexity Sense

- 자주 나오는 시간복잡도 감각:
  - 1차원 DP는 `O(N)` 또는 `O(N^2)`가 자주 나온다
  - 2차원 DP는 `O(N * M)` 형태가 많다
  - 상태 수와 전이 비용을 곱해서 전체 시간복잡도를 본다
- 입력 크기를 볼 때 먼저 판단할 것:
  - `N`이 작은데 경우의 수가 많으면 완전탐색도 가능할 수 있다
  - `N`이 커지면 중복 계산을 줄이는 DP가 필요할 가능성이 높다
  - 메모이제이션과 반복문 DP 중 어느 쪽이 더 안전한지 본다
- 이 topic에서 자주 위험한 비효율:
  - 상태 정의를 너무 크게 잡는 것
  - 같은 전이를 여러 번 중복 계산하는 것
  - 가능한 상태 수를 제대로 계산하지 않고 코드를 먼저 쓰는 것

## Implementation Points

- 코드 작성 시 중요한 포인트:
  - `dp[i]` 또는 `dp[i][j]`가 정확히 무엇을 의미하는지 먼저 한 줄로 정의한다
  - 초기값을 잘못 잡으면 뒤 전부가 무너질 수 있으니 base case를 먼저 확인한다
  - 점화식은 말로 설명한 뒤 코드로 옮긴다
  - 배열 크기와 인덱스 범위를 먼저 정리한다
- 자주 쓰는 자료구조:
  - 배열
  - 2차원 배열
  - 리스트
  - 필요하면 해시 기반 메모이제이션
- 구현 시 실수하기 쉬운 부분:
  - base case 누락
  - 인덱스 오프셋 실수
  - 점화식에서 이전 상태를 잘못 참조하는 실수
  - 최댓값 문제와 경우의 수 문제를 혼동하는 실수
  - 메모리 크기를 충분히 고려하지 않는 실수

## Solving View By Difficulty

- 쉬운 문제에서 주로 보는 것:
  - `dp[i]` 한 줄로 정의 가능한지
  - 이전 1개 또는 2개 상태만 보면 되는지
  - 점화식이 바로 보이는지
- 더 어려운 문제에서 추가로 봐야 하는 것:
  - 상태를 무엇으로 정의해야 중복 없이 표현되는지
  - 차원을 하나 더 추가해야 하는지
  - 현재 보이는 DP보다 더 작은 상태로 압축 가능한지
  - 다른 알고리즘처럼 보여도 실제 핵심이 DP인지
- 조건이 바뀌면 다른 알고리즘으로 넘어가는 지점:
  - 입력이 아주 작으면 백트래킹이나 완전탐색도 가능
  - 탐욕 선택이 항상 맞는 근거가 생기면 그리디로 바뀔 수 있음
  - 연속 구간 조건이 핵심이면 투 포인터나 누적합이 더 자연스러울 수 있음

## Representative Problem Level

이 topic에서는 처음부터 너무 쉬운 문제보다, `중하 ~ 중상` 사이에서 유형을 잘 보여주는 문제를 우선 다룬다.

좋은 대표 문제의 조건은 아래와 같다.

- DP를 왜 써야 하는지 분명히 보인다
- 상태 정의가 중요하다
- 점화식을 직접 세워야 한다
- 시간복잡도를 안 보면 틀릴 수 있다
- 구현은 과도하게 복잡하지 않다

즉, "DP라는 유형을 이해하는 데 도움 되는 문제"를 먼저 다루는 것이 목적이다.

## Common Patterns

DP에서 자주 만나는 대표 패턴은 아래와 같다.

- 1차원 누적 최적화
- 선택 / 비선택
- 계단 오르기 유형
- 연속 부분 수열
- 배낭 문제 기초형
- 문자열 비교 기초형
- 경로 수 세기

이 저장소에서는 처음엔 위 패턴 중 설명력이 좋은 문제를 우선 선택한다.

## Common Mistakes

- 점화식을 외운 형태로만 적용하려고 하는 것
- 상태 정의 없이 바로 코드부터 쓰는 것
- `N` 범위를 보고도 완전탐색을 고집하는 것
- base case를 대충 두는 것
- 배열 크기와 초기값을 문제 조건과 맞추지 않는 것
