# Notes

## Core Signals

- 문제에서 보이는 핵심 신호:
  - 최소 비용을 구한다
  - 현재 집의 색 선택이 이전 집의 색에 직접 영향을 받는다
  - 같은 구조의 계산이 집마다 반복된다
- 문제에서 가장 중요했던 문장:
  - 인접한 두 집은 같은 색으로 칠할 수 없다
- 정답 형태:
  - 최소값

## Candidate Algorithms

- 후보 1: 2차원 DP
- 후보 2: 입력 배열 직접 갱신 방식의 DP
- 후보 3: 완전탐색

## Final Choice

- 최종 선택한 알고리즘: 2차원 bottom-up DP
- 선택 이유:
  - 각 집을 어떤 색으로 칠했는지에 따라 다음 집의 선택 가능 색이 달라진다
  - 현재 집의 최소 비용은 이전 집의 다른 두 색 상태 중 작은 값에 현재 비용을 더하면 된다
  - `N <= 1000`이라 `O(N)`에 색 3개만 확인하는 DP가 가장 자연스럽다
- 사용한 자료구조와 이유:
  - `int[][] costs`
  - `int[][] dp`
  - 집별 색칠 비용과 누적 최소 비용 상태를 분리하면 의미가 더 명확하다

## Complexity Judgment

- 입력 크기 기준 판단:
  - 집 수는 최대 1,000이고 각 집마다 색 3개만 보면 되므로 DP가 매우 안정적이다
- Time Complexity:
  - `O(N)`
- Space Complexity:
  - `O(N)`
- 시간초과 가능성이 있었던 접근:
  - 색칠 순서를 모두 탐색하는 재귀나 브루트포스

## Problem Point

- 이 문제의 핵심 포인트:
  - `dp[i][color]`를 마지막 집의 색까지 포함한 상태로 정의하는 것
  - 현재 색을 고르면 이전 집은 같은 색이 될 수 없으므로, 다른 두 색 상태만 비교하면 된다
- 이 문제를 풀 때 꼭 봐야 했던 조건:
  - 인접한 집끼리만 색이 다르면 된다
  - 마지막 집의 정답은 특정 색이 아니라 세 색 중 최소값이다

## If Conditions Change

- 조건이 달라졌다면 사용할 수 있는 다른 알고리즘:
  - 인접 1칸이 아니라 2칸 이상 제약이 생기면 상태 차원이 더 늘어난 DP
  - 집이 원형으로 이어지면 시작 색을 고정하는 추가 처리
- 지금 풀이가 더 이상 맞지 않게 되는 조건:
  - 색 종류가 많아지거나 제약이 비인접 관계까지 확장되는 경우

## Rejected Ideas

- 시도했지만 버린 접근:
  - 각 집에서 가장 싼 색만 고르는 그리디
  - 모든 색칠 조합을 완전탐색하는 방식
- 버린 이유:
  - 그리디는 현재 집 비용이 싸도 다음 집 선택을 막아 전체 최소를 깨뜨릴 수 있다
  - 완전탐색은 경우의 수가 너무 빠르게 커진다

## Test And Pitfalls

- 예제 테스트:
  - 공식 예제 입력에서 정답 `96`이 나오는지 확인
- 반례 테스트:
  - 현재 집에서 가장 싼 색을 바로 고르면 전체 최소가 깨질 수 있는 경우 확인
- 경계 조건 테스트:
  - `N = 2`
  - 모든 비용이 동일한 큰 입력
- 실수 포인트:
  - 같은 색을 제외하지 않고 이전 집의 세 색 중 최솟값을 그대로 쓰는 실수
  - 마지막 집의 세 색 상태 중 최소를 구해야 하는데 한 색만 반환하는 실수

## Code Comment Points

- 코드에서 설명이 꼭 필요한 부분:
  - `dp[i][color]`의 의미
  - 왜 현재 색과 같은 이전 색은 비교 대상에서 제외하는지
  - 마지막 집에서 세 색 중 최소값을 다시 구하는 이유
- 상태 의미로 주석을 남길 부분:
  - `dp[i][color] = i번째 집까지 칠했을 때, i번째 집 색이 color인 최소 비용`
- 점화식 또는 핵심 전이로 주석을 남길 부분:
  - 빨강 / 초록 / 파랑 각각이 이전 두 색 중 최소를 참조하는 부분
- base case 또는 함정 처리로 주석을 남길 부분:
  - 첫 집 초기화
  - 마지막 줄에서 특정 색이 아니라 세 색 중 최소를 답으로 쓰는 점
