# AGENTS.md

이 파일은 Codex가 이 저장소에서 작업할 때 따라야 할 기본 규칙을 정의한다.

## Project Context

- 이 저장소는 Java 기반 코딩테스트 연습용 저장소다.
- 사용자는 Codex를 일관되고 재사용 가능하게 활용하기 위해 문서와 스킬을 함께 정리하고 있다.
- 모든 제안과 구현은 "코딩테스트 문제를 Java로 안정적으로 푸는 저장소"라는 목적에 맞아야 한다.

## Primary Goals

- Java로 문제 풀이를 작성한다.
- 테스트 가능한 구조를 우선한다.
- 풀이와 함께 시간복잡도, 공간복잡도, 핵심 아이디어를 설명한다.
- 반복 작업은 가능한 경우 skill 형태로 정리한다.
- 문제를 읽을 때 어떤 단서가 어떤 알고리즘 후보로 이어지는지 축적한다.

## Coding Rules

- 기본 언어는 Java다.
- 특별한 이유가 없다면 최신 유행 문법보다 이해하기 쉬운 표준적인 Java 스타일을 우선한다.
- 문제 풀이 코드는 가능한 한 입력, 풀이 로직, 출력의 관심사를 분리한다.
- 알고리즘 핵심 로직은 메서드 단위로 분리해 테스트 가능하게 만든다.
- 불필요한 프레임워크와 과한 추상화는 피한다.
- 이름 규칙과 파일 구조는 `docs/guides/java-naming-conventions.md`를 우선 따른다.
- 실제 문제를 풀 때 전체 흐름은 `docs/guides/problem-solving-workflow.md`를 우선 따른다.
- `Solution.java` 구조는 `docs/templates/solution-template.md`를 참고한다.
- 코드 이해에 꼭 필요한 짧은 설명 주석은 허용한다.
- 특히 상태 의미, base case, 점화식, 반복 순서, 함정 포인트는 짧게 설명해도 된다.
- 단, 코드 한 줄마다 설명을 다는 식의 과한 주석은 피한다.

## Testing Rules

- 새로운 문제 풀이를 추가하면 가능한 한 테스트도 함께 추가한다.
- 테스트 규칙은 `docs/guides/test-conventions.md`를 우선 따른다.
- `SolutionTest.java` 구조는 `docs/templates/solution-test-template.md`를 참고한다.
- 모든 문제는 기본적으로 최소 3개의 테스트를 가진다.
- 예제 입력과 예제 출력이 있으면 이를 테스트 케이스로 우선 반영한다.
- 범위 또는 시간복잡도 관련 테스트를 가능하면 포함한다.
- 함정이나 반례가 보이면 최소 1개 이상 테스트로 반영한다.
- 테스트가 없으면 그 이유를 분명히 설명한다.

## Documentation Rules

- 문제 풀이를 작성할 때 아래 항목을 함께 정리하는 것을 권장한다.
- 문제 요약
- 접근 방법
- 시간복잡도
- 공간복잡도
- 주의할 반례

## Problem Folder Rules

- 문제 하나당 하나의 폴더를 만든다.
- 기본 파일 세트는 `problem.md`, `notes.md`, `Solution.java`, `SolutionTest.java`다.
- `problem.md`에는 문제 핵심 문장, 입력/출력, 제한사항, 헷갈린 표현을 요약한다.
- `notes.md`에는 알고리즘 후보, 선택 이유, 실패한 접근, 반례, 복기 내용을 정리한다.
- `Solution.java`는 실제 풀이 코드다.
- `SolutionTest.java`는 예제와 반례를 검증하는 테스트 파일이다.

## Repository Organization Rules

- 문제는 우선 `topics/topic/problems/problem-slug` 구조로 분류한다.
- 폴더 깊이는 과도하게 늘리지 않는다.
- 각 topic 폴더의 상위 문서에 공통 개념과 중요한 포인트를 정리한다.
- 난이도나 날짜는 폴더보다 문서 메타 정보로 남기는 것을 우선한다.
- 주제가 불분명하면 `implementation`, `data-structure`, `graph`, `dp`, `greedy` 같은 넓은 카테고리부터 사용한다.
- 하나의 문제가 여러 알고리즘과 겹치더라도 대표 topic 하나만 선택한다.
- 관련 알고리즘이나 대체 접근은 `problem.md` 또는 `notes.md`에 기록한다.

## Topic Rules

- 각 topic 폴더에는 `README.md`를 둔다.
- topic의 `README.md`에는 아래 내용을 정리한다.
- 해당 알고리즘의 기본 개념
- 어떤 문제 신호에서 이 topic을 떠올려야 하는지
- 시간복잡도 감각
- 구현 시 중요한 포인트
- 쉬운 문제와 어려운 문제에서 달라지는 관점
- 가능하면 `docs/templates/topic-template.md`의 항목 순서를 유지한다.
- 공통 개념은 topic 상위 문서에 두고, 문제별 특수한 포인트는 각 문제의 `notes.md`에 남긴다.

## Problem Reading Rules

- 문제를 읽을 때 먼저 정답 코드 작성보다 문제 신호를 분류한다.
- 아래 항목을 우선 체크한다.
- 입력 크기
- 그래프/격자 구조 여부
- 최소/최대/최단/경로/경우의 수 같은 목표 문구
- 정렬 이후 처리가 쉬워지는지 여부
- 중복 계산이 발생하는지 여부
- 매 쿼리마다 빠른 응답이 필요한지 여부
- 현재 범위에서 허용 가능한 시간복잡도
- 지금 문제의 조건이 달라지면 풀이가 바뀌는 지점
- `docs/guides/problem-reading-signals.md`의 패턴과 연결해 알고리즘 후보를 제안한다.

## Notes Rules

- `notes.md`는 단순 풀이 요약이 아니라 복기 문서로 작성한다.
- 아래 항목을 가능하면 모두 남긴다.
- 문제에서 보이는 핵심 신호
- 사용 가능한 알고리즘 후보
- 최종 선택한 알고리즘과 이유
- 입력 크기와 시간복잡도 판단
- 이 문제의 핵심 포인트
- 조건이 달라졌다면 바뀌는 풀이
- 반례와 실수 포인트
- 시간초과 가능성이 있었던 접근
- 코드에서 설명 주석이 필요한 포인트

## Collaboration Rules

- 사용자가 먼저 문서 정리를 요청한 경우, 코드보다 문서와 규칙 파일을 우선 정리한다.
- 저장소 구조가 아직 확정되지 않았으면 과도한 구현보다 제안 가능한 최소 구조를 만든다.
- 기존 문서의 톤과 목적을 유지하면서 필요한 내용을 확장한다.
- 사용자의 의도가 불분명하더라도, 코딩테스트 Java 저장소라는 방향에서 합리적으로 가정하고 진행한다.

## Preferred Output Style For Codex

- 설명은 짧고 명확하게 작성한다.
- 가능하면 바로 실행 가능한 다음 단계를 제시한다.
- 파일을 만들거나 수정할 때는 왜 필요한지 한 줄로 설명한다.
- 큰 변경 전에 간단한 계획을 공유한다.
